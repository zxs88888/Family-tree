# Milo 架构迁移影响分析

## 总体评估

当前项目是独立 uni-app H5 应用 + Vercel + Supabase。Milo 底层用的也是 Supabase，但架构模式完全不同。迁移不是简单的配置调整，而是一次**架构重写**。

---

## 用户体感差异

| 维度 | 当前方案 | Milo 方案 | 用户感知 |
|------|----------|-----------|----------|
| 访问方式 | 微信链接 / 浏览器直接打开链接 | 在 Milo 平台内打开，或通过 Milo 生成的链接 | 多了一步"打开 Milo 平台"，但无需自己部署 |
| 登录 | 输入访问码 / 管理员邮箱 Magic Link | **无需手动登录**，Milo 身份自动注入 | 更便捷，省去访问码输入步骤 |
| 域名 | 需要购买独立域名（~50元/年） | 无需购买域名，Milo 提供内置访问地址 | 省去域名费用和配置 |
| 部署维护 | 自行管理 Vercel 部署、域名、Supabase 配额 | Milo 自动分配资源、自动部署 | 零运维 |
| 数据归属 | 自己控制的 Supabase 项目 | Milo 平台分配的 Supabase 实例 | 数据在 Milo 基础设施上 |
| 加载速度 | Vercel CDN 全球加速 | Milo 内置托管 | 差异不大 |

**总结：用户对体感几乎无感知，反而省去了部署、域名和运维成本。**

---

## 对实现的影响

### 影响程度：数据访问层 — 需要重写

| 当前写法 | Milo 写法 | 变动量 |
|----------|-----------|--------|
| `supabase.from('members').select('*').eq('id', id)` | `store.get('members', id)` | 全部替换 |
| `supabase.from('members').insert({...})` | `store.put('members', {...})` | 全部替换 |
| `supabase.from('members').update({...}).eq('id', id)` | `store.put('members', { id, ... })` | 全部替换 |
| `supabase.rpc('validate_access_code', {...})` | **无需**，Milo 身份自动管理 | 删除 |
| `supabase.auth.signInAnonymously()` | **无需** | 删除 |
| `supabase.storage.from('family_photos').upload(...)` | 需改用 Milo 存储 API 或直接 HTTP 上传 | 替换 |
| `supabase.from().order().ilike()` | `store.query()` 不支持 ilike，需在 handler 层 JS 过滤 | 降级或重写 |

所有在 `stores/` 和 `utils/` 里的 Supabase 调用都需要搬到 CLI handler 中。

### 影响程度：前端框架 — 需要重写

| 当前 | Milo |
|------|------|
| uni-app Vue3 + Vite + SPA | `ui/index.html` + 原生 JS |
| Pinia 状态管理 | 移除，状态由 CLI handler 管理 |
| Vue 组件（familyGraph / memberDrawer / ...） | 重写为原生 JS 实现 |
| ECharts 使用 npm 包 | ECharts 可通过 CDN 引入 |
| `@supabase/supabase-js` npm 包 | 移除，换 `callDaemon()` |

**前端框架需要全面重写**：从 uni-app + Vue3 + Vite 转为原生 HTML/JS/CSS。

### 影响程度：认证与权限 — 大幅简化

| 当前 | Milo |
|------|------|
| 访问码验证 + 匿名登录 + Magic Link | Milo Shell 自动管理身份 |
| RLS 策略需防匿名用户 | RLS 只需判断 business logic |
| `admins` 表 + 管理员注册逻辑 | 用户身份由 `getUserId()` 自动获取 |
| 退出/重置流程 | 无需，由 Milo Shell 管理 |

**认证代码可全部删除**，这对安全风控来说反而是改善。

### 影响程度：数据库表结构 — 需微调

当前 `members` 表缺少 `user_id` 列（Milo 强制要求每张表有）。需要：
- `members` 加 `user_id UUID`
- `life_events` 加 `user_id UUID`
- `member_media` 加 `user_id UUID`（已有 `created_by`，但 Milo 端口自动注入的是 `user_id`）
- `families` 表需要 `user_id`（或作为系统表不走 Milo 端口）

其他 RLS 策略仍需保留但调整列名。

### 影响程度：文件存储 — 需要适配

Milo 是否提供 Supabase Storage 同等能力需进一步确认。如果不提供，有两种方案：
1. 用 Milo 的存储端口存文件元数据，实际文件上传到公开 CDN
2. 查看 Milo 是否提供 `storage.bucket` 能力

### 保留不变的部分

- **核心业务逻辑**（配偶双向同步、year_sort 规则、biography 清空逻辑、图标匹配）完全可迁移
- **SQL 迁移脚本**基础部分保留，只需加 `user_id` 列和调整 RLS
- **UI/UX 设计文档**完全适用，视觉风格和交互不变

## 建议

如果目标是 **"让应用尽快可用"**，优先推荐当前方案：
- 当前方案所有代码已经写完、review 完、编译通过
- 只需配置 Supabase + `.env` 就能跑

如果目标是 **"发布为 Milo App 上架市场"**，才值得做迁移。迁移需要约 1 周的框架重写工作，认证和部署部分反而简化了。
