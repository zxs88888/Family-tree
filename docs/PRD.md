# 产品需求文档（PRD）：家族族谱管理 Web 应用

**版本**：V1.8 Final
**发表日期**：2026-06-22
**技术策略**：uni-app（Vue3）开发，优先部署为 H5，预留微信小程序编译能力

---

## 1. 项目背景与目标

### 1.1 背景

传统纸质族谱修订难、更新慢，家族成员（尤其是年轻一代）对自身在庞大家族中的位置缺乏认知。家族已整理出 3 个脉系的 Word 文档，需要快速数字化并串联。

### 1.2 核心目标

1. **极低成本**：利用 Vercel + Supabase 实现近乎零成本的部署与运维
2. **零门槛访问**：以 H5 链接形式在微信家族群传播，长辈点击即看，无需下载安装
3. **自我认知**：登录即高亮显示"我"的直系脉系
4. **高效数据录入**：支持 Excel/CSV 批量导入
5. **生动的生平展示**：每位成员的生平以图文时间线形式呈现，图片为选填项

### 1.3 非目标范围（明确砍掉）

- 社交互动（评论、点赞）
- 视频/音频上传
- 区块链存证
- 自定义家族树布局引擎
- 多家族管理

- **普通成员**：微信打开链接 → 输入访问码 → 找到自己 → 查看家族图谱 → 浏览生平时间线

## 4. 核心业务规则与实现铁律

### 4.1 配偶双向同步（铁律）

当用户将成员 A 的 `spouse_id` 设置为成员 B 时，系统必须同时将成员 B 的 `spouse_id` 更新为成员 A。解除配偶关系时，双方都必须清空。

### 4.2 访问码验证时序（安全关键）

1. 页面加载 → 检查 `localStorage.access_code_verified`
2. 若为 `false` → 阻塞所有数据请求 → 弹出 AccessCodeModal
3. 用户输入 → 调用 `supabase.rpc('validate_access_code')`
4. 验证通过 → **先调用 `supabase.auth.signInAnonymously()`** → 获得 `auth.uid()`
5. 设置 `localStorage.access_code_verified = true`，**然后**调用 `loadAllMembers()`
6. 验证失败 → 提示错误，**不加载任何数据**

**严禁**在访问码验证和匿名登录完成前发起任何 `members` 表的查询。

### 4.3 families 表访问约束 + 家族展示信息获取

前端**永不直接查询** `families` 表。所有 `families` 表数据通过 RPC 获取：
- 访问码验证：`validate_access_code` RPC（SECURITY DEFINER）
- 访问码脱敏提示：`get_access_code_hint` RPC（SECURITY DEFINER）
- 家族展示信息（名称、简介、管理员联系方式）：`get_family_info` RPC（SECURITY DEFINER）

### 4.4 在世成员处理

- 成员录入表单包含"是否在世"开关（默认选中"在世"）
- 选中"在世"：卒年字段隐藏，`death_year` 设为 `NULL`
- 图谱中在世节点边框为绿色虚线，详情页显示"🟢 在世"标签

### 4.5 生平时间线与 biography 展示优先级

| 条件 | 展示内容 |
|------|----------|
| `life_events` 表有数据 | 仅展示时间线，不展示 `members.biography` |
| `life_events` 表无数据 | 降级展示 `members.biography` 纯文本 |

### 4.6 biography 自动清空

- 管理员添加**第一个**时间线事件时，系统自动将 `members.biography` 清空
- 管理员删除**最后一个**时间线事件时，保留 `members.biography` 作为备选但不恢复自动显示
- 提供"从时间线生成生平简介"按钮（仅用于导出/打印，不写回 `biography` 字段）

### 4.7 时间线事件 year_sort 规则

| 年份格式 | 自动填充 year_sort | 说明 |
|----------|-------------------|------|
| `1965` | `1965` | 纯数字年份 |
| `约1940年` | `1940` | 提取 4 位数字 |
| `1930年代` | `1935` | 取年代中值 |
| `80年代` | `1985` | 默认 20 世纪中值 |
| `清光绪年间` | `NULL` | 不做自动转换，用户手动填写 |

### 4.8 删除事件

删除事件时必须同步：
1. 查询事件关联的 media 记录
2. 删除 Supabase Storage 中的对应文件
3. 删除 `member_media` 记录
4. 删除 `life_events` 记录
5. 清空事件缓存

### 4.9 标签图标匹配

- 按**包含匹配**（`includes`）识别标签关键词
- 同时匹配多个关键词时取**最长匹配**
- 标签为空时不显示图标
- 未匹配关键词统一使用 📌

| 标签 | 图标 |
|------|------|
| 出生 | 🎂 |
| 求学/毕业 | 📚 |
| 工作/入职 | 💼 |
| 婚姻/结婚 | 💍 |
| 居住/迁居 | 🏠 |
| 成就/获奖 | 🏆 |
| 参军/退伍 | 🎖️ |
| 移民/出国 | ✈️ |

## 5. 详细功能需求

### 5.1 入口与认证

#### 5.1.1 访问方式

微信内点击 H5 链接，或复制链接到浏览器打开。

#### 5.1.2 管理员认证（邮箱 Magic Link）

- 管理员输入邮箱 → Supabase 发送登录链接 → 点击链接即登录
- 登录后获得 `auth.uid()`，拥有增删改权限

#### 5.1.3 管理员自动注册

1. 用户通过 Supabase Auth 邮箱登录后，检查 `admins` 表
2. 若该 `family_id` **不存在任何**管理员记录 → 当前用户自动成为超级管理员
3. 若**已存在**管理员记录 → 当前用户不自动成为管理员，仅作为普通成员

#### 5.1.4 普通成员认证（家族访问码）

- 首页输入框输入静态的、可更换的"家族访问码"
- 验证通过后调用 `supabase.auth.signInAnonymously()` 获得临时 `auth.uid()`
- RLS 要求 `auth.role() = 'authenticated'`

#### 5.1.5 退出/重置

点击右上角"退出/重置"（↻）图标：
1. 清空所有 LocalStorage 缓存（`my_member_id`、`access_code_verified`）
2. 调用 `supabase.auth.signOut()`
3. 页面强制刷新，回到初始状态
4. 提示"已退出，请重新输入访问码或登录"

#### 5.1.6 首次登录"选我"流程

1. 登录/验证访问码后，弹窗引导用户"在家族中找到自己"
2. 使用搜索框模糊匹配，展示前 5 位候选人
3. 候选信息格式："张三（父亲：张二）"
4. 用户确认后将 `member_id` 存入 LocalStorage
5. 后续自动读取，高亮展示"我的脉系"
6. 右上角提供"切换身份"入口

### 5.2 首页：家族概览与图谱

- 展示家族名称、家族简介
- 家族总人数、在世人数、覆盖世代数
- ECharts Graph 关系图主区域
- 图谱左上角视图切换器：🧬 脉系视图 / 🌳 全局视图

### 5.3 核心交互：家谱关系图

#### 5.3.1 默认视图

- 进入后默认展示"我的脉系"视图
- 基于 LocalStorage 中的 `member_id`

#### 5.3.2 视图切换

- **脉系视图**：以"我"为中心，展开上下 3 代，金色血脉高亮（BFS 全路径标记）
- **全局视图**：以家族首位先祖为根，展开 `depth=5`（>300 人时默认 `depth=3`），无金色高亮，所有连线灰色
- **禁用状态**：`myMemberId === null` 时脉系视图不可用，点击提示"请先在家族中选择您自己"

#### 5.3.3 按需懒加载

- 初始只渲染以"我"为中心的上下 3 代成员
- 每个成员节点旁显示"展开更多"按钮（存在未加载子嗣时）
- 点击 → 内存中扩展 BFS depth → 增量更新图谱（不发起新请求）

#### 5.3.4 节点交互

- 点击成员节点 → 底部弹出抽屉（Drawer），显示成员详情（含生平时间线）
- 所有操作仅依赖单击，禁止长按/右键菜单

#### 5.3.5 金色血脉与渲染深度一致性

- BFS 标记不受 `depth` 限制，标记所有直系祖先和后代
- 图谱仅展示以"我"为中心上下 3 代节点
- **MVP**：标记但未渲染的血脉连线末端使用 ECharts `edgeSymbol: ['none', 'arrow']` 显示金色箭头，暗示血脉延续
- **P2**：自定义 SVG 覆盖层显示"…"或"更多祖先"文字提示（ECharts 原生 edgeLabel 位置受限，MVP 暂不做）

#### 5.3.6 在世成员视觉标识

- 图谱节点边框：在世绿色虚线（`#27AE60`），已故灰色实线
- 详情页：在世显示"🟢 在世"标签，已故显示生卒年范围

### 5.4 数据录入与冷启动

#### 5.4.1 三步建谱向导

1. 录入"自己"（姓名、性别、是否在世）
2. 家族管理员联系方式（选填）
3. 录入"父亲"（可选录入母亲）
4. 录入"祖父"（可选录入祖母）
- 系统自动生成三代骨架树（含配偶关系）

#### 5.4.2 后续录入

在树节点上点击"添加子女/配偶"，直接补充。

### 5.5 生平时间线

#### 5.5.1 事件编辑（管理员）

**事件字段**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 年份 | ✅ | 支持自由格式："1965"、"约1940年"、"1930年代"等 |
| 年份排序值 | 系统自动生成，可手动修正 | 按规则自动填充 |
| 事件标签 | ❌ | 自由文本，提供常用标签快速选择按钮 |
| 事件标题 | ✅ | ≤ 50 字 |
| 事件地点 | ❌ | |
| 详细描述 | ❌ | ≤ 2000 字，支持换行 |
| 图片 | ❌ | 选填多张，每张可加说明文字（≤ 50 字），支持拖拽排序 |

**常用标签快速选择按钮**：
[出生] [求学] [工作] [婚姻] [居住] [成就]

点击后填入对应标签，允许用户继续编辑修改。

**事件列表展示**（编辑页面底部）：
- 按 `year_sort` 降序排列
- 每条显示：年份展示 + 标签 + 标题 + 图片数量徽标
- 每条右侧有"编辑"和"删除"按钮

#### 5.5.2 时间线展示（普通成员）

**事件卡片结构**：

```
┌─────────────────────────────────────────────┐
│ 1985  📚 求学 · 北京大学毕业        [展开 ▾] │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
│  考入北大物理系，获学士学位                  │
│  地点：北京                                  │
│  [📷 毕业照] [📷 校园合影]  ← 有图事件     │
└─────────────────────────────────────────────┘
```

**默认折叠**：仅显示年份、标签图标、标题和"展开 ▾"
**展开后**：显示完整描述、地点、图片缩略图（前 3 张）
**无图事件**：图片区域留空，不显示任何提示
**图片缩略图**：展开后显示前 3 张（横向 80x80），超过 3 张显示 "+N"
**点击缩略图** → 全屏图库预览

#### 5.5.3 图库视图

- 入口：时间线底部"查看全部 X 张照片"
- MVP 方案：使用 `uni.previewImage` + `indicator` 参数显示编号
- 自定义上下文浮层（年份+标题）列为 P2
- 空图库："这位成员暂无历史照片，欢迎补充。"

#### 5.5.4 图片上传说明

- 图片为**选填项**，不强制要求
- 许多历史阶段没有照片留存，这完全正常
- 录入原则：文字信息是历史记录的核心，图片是"锦上添花"

### 5.6 Excel/CSV 批量导入

#### 5.6.1 模板列定义

| 列名 | 必填 | 类型 | 说明 |
|------|------|------|------|
| `姓名` | ✅ | 文本 | 成员全名 |
| `性别` | ❌ | 数字 | 1=男，2=女，空=未知 |
| `父亲` | ❌ | 文本 | 优先文件内查找，再查数据库 |
| `母亲` | ❌ | 文本 | 同上 |
| `配偶` | ❌ | 文本 | 同上 |
| `生年` | ❌ | 数字 | 4 位数字年份 |
| `卒年` | ❌ | 数字 | 在世留空 |
| `生平简介` | ❌ | 文本 | 纯文本，导入后可在系统中拆分为时间线 |
| `时间线` | ❌ | 文本 | 结构化事件，`[年份] 标签：标题 \|\| 描述 \|\| 地点`，多事件用 `\|\|\|` 分隔 |

**导入优先级**：时间线列优先，`生平简介` 作为备份写入 `members.biography`。

#### 5.6.2 时间线列格式

| 格式 | 说明 |
|------|------|
| `[1985] 求学：北大毕业 \|\| 考入北大物理系 \\\| 获学士学位 \|\| 北京` | 单个事件，`\|\|` 分隔字段，`\\\|` 转义管道符 |
| `事件1 \|\|\| 事件2` | 多个事件用 `\|\|\|` 分隔 |

#### 5.6.3 校验规则

| 校验项 | 规则 |
|--------|------|
| 姓名非空 | 每行必须有姓名 |
| 性别合法性 | 值必须为 1、2 或空 |
| 年份合法性 | 4 位数字，范围 1800-2026 |
| 生卒年逻辑 | 卒年 ≥ 生年 |
| 引用存在性 | 文件内查找 → 数据库查找 |
| 姓名唯一性 | 同一文件中姓名不可重复 |
| 循环依赖 | 父子关系不得构成循环 |
| 时间线格式 | 符合 `[年份] 标签：标题 \|\| 描述 \|\| 地点` |

**校验失败处理**：全部回滚，不创建任何数据。

#### 5.6.4 导入后引导

- 成功统计：导入人数、事件数、失败数
- 自动建立关系统计
- "待完善成员"（孤立成员）筛选器：`father_id IS NULL AND mother_id IS NULL AND spouse_id IS NULL`

#### 5.6.5 导入性能指标

- 单次上限：10,000 条记录
- 解析速度：10,000 条 < 3 秒
- 写入速度：10,000 条 < 10 秒

## 6. 隐私、安全与权限

### 6.1 双轨制权限模型

| 维度 | 管理员 | 普通成员 |
|------|--------|----------|
| 认证方式 | 邮箱 Magic Link | 家族访问码 + 匿名登录 |
| 数据查看 | 全部 | 全部（只读） |
| 数据编辑 | 全部 | 无 |
| 数据删除 | 全部 | 无 |
| 管理员管理 | 是 | 否 |

### 6.2 安全要求

- 强制 HTTPS
- RLS 要求 `auth.role() = 'authenticated'`
- 前端不直接查询 `families` 表
- 访问码通过 SECURITY DEFINER RPC 验证
- 删除事件时同步删除 Storage 文件
- Vercel WAF

### 6.3 域名策略

- 购买独立域名（.top/.xyz/.site，约 30-50 元/年）
- 使用二级域名（如 `tree.xxx.top`）
- DNS 添加 CNAME 到 `cname.vercel-dns.com`
